Documentación del Proyecto

Descripción general


El Sistema de Gestión de Procesos es una plataforma web diseñada para que las empresas puedan documentar, administrar y mantener actualizados sus flujos de trabajo internos. Conecta a los distintos colaboradores de una organización en un entorno unificado donde se pueden gestionar procesos, actividades, roles y decisiones de manera estructurada.


La aplicación ofrece un entorno seguro, colaborativo e intuitivo para la creación de procesos empresariales, facilitando la asignación de roles, la trazabilidad de cambios y la visualización gráfica de los flujos. El sistema contempla dos niveles de usuario principales:


  • Administrador de empresa: Puede registrar la empresa en el sistema, gestionar usuarios, crear y administrar procesos, definir roles, eliminar elementos y garantizar la consistencia de la información.
  • Usuario autorizado: Puede crear, editar y consultar procesos, actividades, arcos y gateways, así como aportar a la documentación según los permisos asignados.

El sistema cuenta con autenticación y autorización para controlar accesos, registro de historial de cambios y funcionalidades de búsqueda y filtrado. De esta manera, las organizaciones pueden asegurar que sus procesos estén correctamente documentados, actualizados y disponibles para todos sus colaboradores.


Definiciones clave

  • Proceso: Conjunto estructurado de actividades que representan un flujo de trabajo dentro de la empresa. Puede estar en estado borrador o publicado.
  • Actividad: Tarea individual dentro de un proceso. Tiene nombre, tipo, descripción y un rol responsable asociado.
  • Arco: Conexión que enlaza actividades y/o gateways, definiendo el orden y la secuencia en que se ejecuta el proceso.
  • Gateway: Punto de decisión dentro del proceso, que puede ser exclusivo, paralelo o inclusivo, y define cómo se ramifica el flujo.
  • Rol de proceso: Función definida dentro de la empresa (ej. Analista, Supervisor, Auditor) a la que se asignan actividades, en lugar de asociarlas a personas específicas.

Requerimientos Funcionales


RF1. Autenticación y gestión de usuarios:

  • RF1.1: El sistema debe permitir registrar una empresa solicitando nombre, NIT y correo de contacto, generando un administrador inicial.
  • RF1.2: El sistema debe permitir a los usuarios iniciar sesión con credenciales válidas, restringiendo el acceso a la empresa correspondiente.
  • RF1.3: El sistema debe permitir a los usuarios cerrar sesión de manera segura.

RF2. Gestión de procesos:

  • RF2.1: El sistema debe permitir que un usuario autorizado cree procesos registrando nombre, descripción y categoría.
  • RF2.2: El sistema debe permitir que un usuario autorizado edite procesos, guardando un historial de cambios.
  • RF2.3: El sistema debe permitir que un administrador elimine procesos, manteniéndolos en estado inactivo para conservar trazabilidad.
  • RF2.4: El sistema debe permitir a los usuarios consultar procesos existentes mediante búsqueda y filtros.

RF3. Gestión de actividades:

  • RF3.1: El sistema debe permitir que un usuario autorizado cree actividades definiendo nombre, tipo, descripción y rol responsable.
  • RF3.2: El sistema debe permitir que un usuario autorizado edite actividades, registrando las modificaciones en el historial del proceso.
  • RF3.3: El sistema debe permitir que un administrador elimine actividades, ajustando automáticamente el flujo del proceso.

RF4. Gestión de arcos:

  • RF4.1: El sistema debe permitir que un usuario autorizado cree arcos definiendo origen y destino entre actividades o gateways.
  • RF4.2: El sistema debe permitir que un usuario autorizado edite arcos existentes.
  • RF4.3: El sistema debe permitir que un administrador elimine arcos, ajustando automáticamente el diagrama.

RF5. Gestión de gateways:

  • RF5.1: El sistema debe permitir que un usuario autorizado cree gateways de tipo exclusivo, paralelo o inclusivo.
  • RF5.2: El sistema debe permitir que un usuario autorizado edite gateways, actualizando condiciones de salida.
  • RF5.3: El sistema debe permitir que un administrador elimine gateways, asegurando la consistencia del flujo.

RF6. Gestión de roles de proceso:

  • RF6.1: El sistema debe permitir que un usuario autorizado cree roles de proceso (ejemplo: Analista, Supervisor, Auditor).
  • RF6.2: El sistema debe permitir que un usuario autorizado edite roles de proceso sin afectar el historial de los procesos.
  • RF6.3: El sistema debe permitir que un administrador elimine roles de proceso, validando que no estén asignados a actividades.
  • RF6.4: El sistema debe permitir que cualquier usuario consulte los roles de proceso disponibles y en qué procesos se utilizan.

RF7. Asignación de roles de cuenta:

  • RF7.1: El sistema debe permitir la creación y asignación de roles a los procesos de la empresa.

Requerimientos No Funcionales


RNF1. Seguridad y privacidad:

  • RNF1.1: El sistema debe contar con autenticación mediante correo y contraseña.
  • RNF1.2: El sistema debe implementar autorización por rol y empresa, de manera que los usuarios solo accedan a la información de su organización.
  • RNF1.3: El sistema debe permitir configurar políticas de contraseñas con requisitos mínimos de longitud y complejidad.

RNF2. Disponibilidad y confiabilidad:

  • RNF2.1: El sistema debe garantizar una disponibilidad mínima del servicio.

RNF3. Rendimiento:

  • RNF3.1: El sistema debe responder en tiempos adecuados en operaciones de consulta.
  • RNF3.2: El sistema debe garantizar un tiempo aceptable en operaciones de escritura (crear, editar, eliminar).
  • RNF3.3: El sistema debe renderizar diagramas de procesos en un tiempo razonable incluso con un número elevado de elementos.
  • RNF3.4: El sistema debe someterse a pruebas de rendimiento y carga utilizando JMeter, cumpliendo con tiempos de respuesta y estabilidad bajo escenarios de estrés.

RNF4. Escalabilidad:

  • RNF4.1: El sistema debe ser escalable para soportar un crecimiento en el número de empresas y usuarios concurrentes.
  • RNF4.2: El sistema debe mantener un desempeño adecuado aun con grandes volúmenes de procesos y actividades.

RNF5. Mantenibilidad:

  • RNF5.1: El sistema debe contar con pruebas unitarias y de integración para garantizar la calidad del código.
  • RNF5.2: El sistema debe estar desarrollado siguiendo estándares de codificación que faciliten la lectura y mantenimiento.
  • RNF5.3: El sistema debe cumplir con métricas de calidad de código evaluadas mediante SonarQube, alcanzando como mínimo un nivel aceptable en cobertura de pruebas, duplicación de código y complejidad.

RNF6. Usabilidad:

  • RNF6.1: El sistema debe contar con una interfaz intuitiva y fácil de usar para usuarios no técnicos.

RNF7. Integridad y consistencia de datos:

  • RNF7.1: El sistema debe mantener la consistencia de los datos en operaciones de creación, edición y eliminación.
  • RNF7.2: El sistema debe prevenir la eliminación de elementos que estén siendo utilizados en otros procesos (ejemplo: roles asignados a actividades).
  • RNF7.3: El sistema debe conservar un historial inalterable de cambios realizados.

RNF8. Monitoreo y registro:

  • RNF8.1: El sistema debe registrar métricas de uso y rendimiento.
  • RNF8.2: El sistema debe generar alertas ante fallos o errores recurrentes.

RNF9. Cumplimiento legal:

  • RNF9.1: El sistema debe cumplir con las normativas locales de protección de datos personales.
  • RNF9.2: El sistema debe contar con políticas de privacidad y términos de uso aceptados por los usuarios.

RNF10. Portabilidad y compatibilidad:

  • RNF10.1: El sistema debe ser compatible con los navegadores web más utilizados (ejemplo: Google Chrome, Mozilla Firefox, Microsoft Edge).

Diagrama de arquitectura


El diagrama de arquitectura del sistema ArriendaTuFinca describe la organización de sus componentes y la interacción entre las distintas capas.


  • Capa Cliente: Usuarios arrendadores y arrendatarios interactúan con la aplicación a través de una interfaz web desarrollada en Angular.
  • Capa de Entrada: Gestiona las solicitudes HTTP(S) mediante Ingress y control de CORS, garantizando la seguridad y el manejo adecuado de peticiones REST/JSON.
  • Capa de Seguridad: Implementada con Spring Security y autenticación JWT, se encarga de proteger los accesos a los servicios internos del sistema.
  • Capa de Aplicación: Desarrollada con Spring Boot y JPA, contiene los servicios de Autenticación, Propiedades, Reservas y Pagos, encargados de la lógica de negocio.
  • Capa de Datos: Administra la persistencia de información en la base de datos, almacenando propiedades, usuarios, reservas y pagos.
  • Integración con módulo de pagos: El servicio de pagos interactúa con el sistema externo para procesar transacciones y devolver los resultados al sistema.

    Diagrama de Contexto


    Este diagrama representa cómo los usuarios arrendadores y arrendatarios se comunican con la plataforma para publicar, buscar y reservar propiedades, así como gestionar disponibilidad, precios y calificaciones.


    También se visualiza la relación con el módulo de pagos para procesar transacciones y con la base de datos, donde se almacenan y consultan las propiedades, usuarios, reservas y pagos, garantizando el flujo de información necesario para el funcionamiento del sistema.


    Diagrama de Contenedor


    Este diagrama de contenedor muestra la forma en que interactúan sus diferentes componentes y actores.


    El sistema cuenta con dos tipos de usuarios: arrendador y administrador, quienes acceden desde un navegador web a través de conexiones seguras HTTP/S.


    La interfaz está desarrollada con Thymeleaf Templates, lo que permite mostrar dinámicamente la información y procesar las interacciones del usuario.


    Las solicitudes se envían a la aplicación Spring Boot, que concentra los controladores, servicios y mecanismos de seguridad. Esta aplicación se comunica con dos elementos principales:


  • Una base de datos relacional (MySQL), mediante JPA, para almacenar y gestionar la información de manera persistente.
  • Un servidor SMTP, encargado del envío de correos electrónicos de activación, lo que garantiza la validación y activación de cuentas.

    • Diagrama de Componentes


    Este diagrama esta desarrollado bajo un enfoque de capas que permite mantener una separación clara entre la presentación, la lógica de negocio y la persistencia de datos, lo que facilita su mantenimiento, escalabilidad y seguridad.


    En la parte superior se encuentra la capa de presentación, implementada con Thymeleaf y templates HTML, encargada de mostrar la información al usuario y recibir sus interacciones. Esta capa se conecta con los controladores web, los cuales procesan las solicitudes y sirven como puente hacia la capa de lógica de negocio.


    La capa de lógica de negocio es el núcleo del sistema. Aquí se ubican los servicios que gestionan las funcionalidades clave: la administración de usuarios, la autenticación y seguridad con Spring Security, y la gestión de propiedades. Esta capa se comunica tanto con el módulo de persistencia de datos como con las integraciones externas.


    La persistencia de datos se implementa con Spring Data JPA, lo que permite gestionar la información de manera eficiente a través de repositorios que trabajan con entidades (Entities) y DTOs para transportar datos de forma optimizada. La información se almacena en una base de datos relacional, que en nuestro proyecto va a ser MySQL.


    Por último, el sistema cuenta con integraciones externas, como el servicio de correo electrónico (SMTP) y otras API que amplían las funcionalidades de la aplicación.


    Diagrama de Código


    Este diagrama muestra la estructura de código de la aplicación, organizada siguiendo el patrón MVC (Modelo-Vista-Controlador) y evidenciando la relación entre controladores, servicios, repositorios y entidades. En la capa de controladores se encuentran CuentaController, ArrendoController, FincaController y CalificacionController, encargados de recibir las solicitudes de los usuarios y delegarlas a sus respectivos servicios: CuentaService, ArrendoService, FincaService y CalificacionService.


    Cada servicio se comunica con su correspondiente repositorio —UsuarioRepository, ArrendoRepository, FincaRepository y CalificacionRepository—, que son responsables de interactuar con la base de datos.


    Las entidades principales del sistema son:


    • Usuario: con atributos como id, nombre, email y password.
    • Arriendo: con información como id, fechaInicio, fechaFin y precio.
    • Finca: con id, dirección y precio.
    • Calificación: con id, puntaje y comentario.

    Las relaciones entre entidades muestran que:


    • Un usuario puede tener múltiples arriendos y múltiples fincas.
    • Una finca puede tener varios arriendos y varias calificaciones.
    • Cada arriendo está asociado a un usuario y a una finca.

    Diagrama de Clases


    Este diagrama de clases representa la estructura y relaciones principales del sistema "Arrienda tu Finca".


    • Usuario: Clase central que almacena la información personal (nombre, apellido, correo, teléfono, rol) y métodos para gestionar la cuenta (crear, editar, ver perfil, activar, autenticar). Se asocia con múltiples Finca, Solicitud y Calificación.
    • Rol (ENUM): Define los tipos de usuario: ARRENDADOR y ARRENDATARIO.
    • Finca: Contiene datos de una propiedad (título, descripción, ubicación, precio, disponibilidad) y métodos para su administración (publicar, editar, desactivar, marcar disponible, subir fotos, ver fincas). Relacionada con un propietario (Usuario) y con Calificación.
    • Solicitud: Representa una petición de arriendo de una finca, con atributos como fecha, estado y mensaje, y métodos para gestionarla (aceptar, rechazar, ver solicitudes).
    • Pago: Registra los pagos asociados a solicitudes, con detalles como monto, fecha y método de pago, y el método realizarPago.
    • Calificación: Permite evaluar la experiencia con puntaje, comentario y fecha, vinculando al evaluador, evaluado y la finca correspondiente.

    En conjunto, el diagrama muestra cómo las entidades del sistema se conectan para permitir la publicación de fincas, el envío y gestión de solicitudes, la realización de pagos y la calificación de experiencias.


    Diagrama de despliegue


    El diagrama de despliegue muestra la infraestructura física o virtual donde se ejecutará la aplicación. En este caso, la aplicación se ejecuta en una máquina virtual que contiene un servidor Apache, Kubernetes (K8s) para orquestación de contenedores, la aplicación web de Spring Boot y la base de datos JPA.

    Base de datos

    Modelo Entidad-Relación (ER)


    El siguiente diagrama Entidad-Relación (ER) representa la estructura de la base de datos propuesta para el sistema, mostrando las entidades principales, sus atributos y las relaciones entre ellas. Este modelo permite visualizar cómo se organiza y conecta la información, sirviendo como base para el diseño e implementación del sistema. En el diagrama, PK (Primary Key) es una llave primaria, identificador único de cada registro en la tabla. No se repite y no puede ser nulo y FK (Foreign Key) es una llave foránea, campo que establece una relación con la clave primaria de otra tabla para mantener la integridad referencial.


    Diccionario de datos


    Entidad: ROL

    • PK: nombre : STRING
    • Descripción: Tipo de rol del usuario (arrendador, arrentatario).

    Entidad: USUARIO

    • PK: id : STRING
    • Atributos: nombre (STRING), apellido (STRING), correo (STRING), telefono (STRING), rol (STRING - FK a ROL.nombre).

    Entidad: FINCA

    • PK: id : LONG
    • Atributos: titulo (STRING), descripcion (STRING), ubicacion (STRING), precio (LONG), disponible (BOOLEAN), propietaria (STRING - FK a USUARIO.id).

    Entidad: SOLICITUD

    • PK: id : LONG
    • Atributos: fecha (LocalDate), estado (STRING), mensaje (STRING), finca (LONG - FK a FINCA.id), arrendador (STRING - FK a USUARIO.id).

    Entidad: PAGO

    • PK: id : LONG
    • Atributos: monto (LONG), fecha (LocalDate), metodo (STRING), solicitud (LONG - FK a SOLICITUD.id).

    Entidad: CALIFICACION

    • PK: id : LONG
    • Atributos: puntaje (INT), comentario (STRING), fecha (LocalDate), finca (LONG - FK a FINCA.id), evaluador (STRING - FK a USUARIO.id), evaluado (STRING - FK a USUARIO.id).

    Relaciones:

    • Un ROL puede asignarse a muchos USUARIOS, pero cada USUARIO tiene un solo ROL.
    • Un USUARIO puede poseer varias FINCAS.
    • Una FINCA puede recibir múltiples SOLICITUDES.
    • Una SOLICITUD puede generar uno o varios PAGOS.
    • Un USUARIO puede emitir múltiples CALIFICACIONES.
    • Una FINCA puede recibir múltiples CALIFICACIONES.
